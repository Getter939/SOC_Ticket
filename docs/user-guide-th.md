# คู่มือการใช้งานระบบบริหารจัดการเหตุการณ์ความปลอดภัย (SOC Ticketing System)

คู่มือฉบับนี้จัดทำขึ้นสำหรับผู้ใช้งานระบบบริหารจัดการเหตุการณ์ความปลอดภัยของ NT โดยแบ่งเนื้อหาออกเป็น 3 ส่วนหลัก ได้แก่ ภาพรวมระบบสำหรับผู้ใช้งานทุกบทบาท คู่มือสำหรับนักวิเคราะห์ SOC และคู่มือสำหรับผู้บริหาร

---

# 1. ภาพรวมระบบ (System Overview)

## ระบบนี้คืออะไร

ระบบบริหารจัดการเหตุการณ์ความปลอดภัย (SOC Ticketing System) คือเครื่องมือกลางที่ทีมศูนย์ปฏิบัติการความปลอดภัย (Security Operations Center — SOC) ของ NT ใช้ในการรับแจ้ง บันทึก ติดตาม และปิดเหตุการณ์ด้านความปลอดภัยทางไซเบอร์อย่างเป็นระบบ

ก่อนหน้านี้การรับมือเหตุการณ์อาจกระจัดกระจายอยู่ตามอีเมล โทรศัพท์ หรือการจดบันทึกส่วนบุคคล ทำให้ติดตามได้ยากว่าใครรับผิดชอบเรื่องใด ดำเนินการถึงขั้นตอนไหน และใช้เวลาไปเท่าใด ระบบนี้แก้ปัญหาดังกล่าวโดย:

- รวมทุกเหตุการณ์ไว้ในที่เดียวในรูปแบบของ **ตั๋วงาน (Ticket)** ที่มีหมายเลขอ้างอิงเฉพาะ
- กำหนดขั้นตอนการทำงานที่ชัดเจน ว่าแต่ละเหตุการณ์ต้องผ่านมือใครบ้างก่อนปิดได้
- บันทึกประวัติการดำเนินการทุกขั้นตอนโดยอัตโนมัติ เพื่อการตรวจสอบย้อนหลัง
- แจ้งเตือนผู้เกี่ยวข้องทางอีเมลเมื่อมีงานที่ต้องดำเนินการ
- สรุปสถิติภาพรวมให้ผู้บริหารเห็นประสิทธิภาพการทำงานของทีม

## การเข้าใช้งานและการเข้าสู่ระบบ

1. เปิดเว็บเบราว์เซอร์ แล้วเข้าไปยังที่อยู่ของระบบที่ผู้ดูแลระบบแจ้งให้คุณทราบ
2. ระบบจะแสดงหน้าเข้าสู่ระบบ ให้กรอก **ชื่อผู้ใช้ (Username)** และ **รหัสผ่าน (Password)** ที่คุณได้รับ
3. กดปุ่มเข้าสู่ระบบ

> ⚠️ บัญชีผู้ใช้งานทุกบัญชีถูกสร้างและกำหนดบทบาทโดยผู้ดูแลระบบของ SOC คุณไม่สามารถสมัครใช้งานด้วยตนเองได้ หากยังไม่มีบัญชีหรือลืมรหัสผ่าน กรุณาติดต่อทีม SOC

[ภาพหน้าจอ: หน้าเข้าสู่ระบบ แสดงช่องกรอกชื่อผู้ใช้ รหัสผ่าน และปุ่มเข้าสู่ระบบ]

เมื่อเข้าสู่ระบบสำเร็จ คุณจะเห็นเมนูทางด้านซ้ายมือ เมนูที่ปรากฏจะแตกต่างกันไปตามบทบาทของคุณ ระบบจะแสดงเฉพาะเมนูที่คุณมีสิทธิ์ใช้งานเท่านั้น

[ภาพหน้าจอ: หน้าจอหลักหลังเข้าสู่ระบบ แสดงแถบเมนูด้านซ้ายและพื้นที่แสดงผลหลัก]

## แนวคิดหลักที่ผู้ใช้งานทุกคนต้องเข้าใจ

ก่อนเริ่มใช้งาน คุณควรทำความเข้าใจคำสำคัญ 4 คำต่อไปนี้

### ตั๋วงาน (Ticket)

**ตั๋วงาน (Ticket)** คือบันทึกเหตุการณ์ความปลอดภัยหนึ่งเรื่อง เปรียบเสมือนแฟ้มงานหนึ่งแฟ้มที่รวบรวมข้อมูลทั้งหมดของเหตุการณ์นั้นไว้ เช่น รายละเอียดเหตุการณ์ ระดับความรุนแรง ทรัพย์สินที่ได้รับผลกระทบ ผู้รับผิดชอบ และประวัติการดำเนินการ ตั๋วงานทุกใบมีหมายเลขอ้างอิงเฉพาะ (เช่น `260601`) ที่ระบบสร้างให้อัตโนมัติ ใช้สำหรับอ้างอิงและค้นหา

### สถานะ (Status / State)

**สถานะ (Status)** คือขั้นตอนปัจจุบันที่ตั๋วงานอยู่ เช่น เพิ่งแจ้งเหตุใหม่ กำลังรอผู้ดูแลระบบจัดการ หรือปิดงานแล้ว สถานะจะเปลี่ยนไปตามลำดับขั้นตอนการทำงาน และเปลี่ยนได้เฉพาะตามเส้นทางที่ระบบกำหนดไว้เท่านั้น (ดูตารางในหัวข้อถัดไป)

### บทบาท (Role)

**บทบาท (Role)** คือหน้าที่และสิทธิ์ของคุณในระบบ ระบบกำหนดบทบาทไว้ดังนี้

| บทบาท | คำอธิบายโดยย่อ |
|--------|----------------|
| นักวิเคราะห์ SOC ระดับ 1 (SOC Analyst Tier 1) | ผู้เปิดตั๋วงาน คัดแยกประเภทเหตุการณ์ ตรวจสอบ และดำเนินการฝั่งต้นทางทั้งหมด |
| นักวิเคราะห์ SOC ระดับ 2 (SOC Analyst Tier 2) | ผู้รับเรื่องที่ถูกส่งต่อขึ้นมา เพื่อวินิจฉัยเหตุการณ์ที่ระดับ 1 ยังไม่แน่ใจ |
| ผู้จัดการ SOC (SOC Manager) | ผู้บริหารที่ตรวจสอบและอนุมัติปิดงานเหตุการณ์สำคัญ และดูภาพรวมสถิติ |
| ผู้ดูแลระบบ (System Admin) | ผู้รับผิดชอบเข้าควบคุมและแก้ไขที่ตัวระบบ/อุปกรณ์ที่ถูกโจมตี |
| เจ้าของระบบ (System Owner) | เจ้าของหรือหน่วยงานที่ดูแลระบบที่ได้รับผลกระทบ รับทราบสถานะแบบอ่านอย่างเดียว |

> ระบบจำกัดสิทธิ์อย่างเข้มงวด การกระทำแต่ละอย่าง (เช่น การเปิดตั๋ว การปิดงาน การอนุมัติ) ทำได้เฉพาะบทบาทที่ได้รับมอบหมายเท่านั้น หากคุณไม่เห็นปุ่มหรือเมนูใด แสดงว่าการกระทำนั้นไม่ใช่หน้าที่ของบทบาทคุณ

### ข้อตกลงระดับบริการ (SLA)

**ข้อตกลงระดับบริการ (Service Level Agreement — SLA)** คือกรอบเวลามาตรฐานที่ทีม SOC ต้องตอบสนองต่อเหตุการณ์ ระบบกำหนดกรอบเวลาไว้ที่ **4 ชั่วโมง** นับจากเวลาที่ตรวจพบเหตุการณ์ ระบบจะคำนวณกำหนดเส้นตาย (SLA Deadline) ให้อัตโนมัติ และแสดงเตือนเมื่อใกล้ครบกำหนดหรือเกินกำหนด

## แผนผังสถานะของตั๋วงาน

ตั๋วงานทุกใบจะเดินตามเส้นทางสถานะที่กำหนดไว้ ไม่สามารถข้ามขั้นหรือย้อนกลับนอกเส้นทางได้ ตารางต่อไปนี้สรุปสถานะทั้งหมด

### ตารางสถานะทั้งหมด

| สถานะ (รหัส) | ชื่อภาษาไทย | ความหมาย | เป็นสถานะปิดงาน |
|--------------|------------|----------|:----------------:|
| NEW | แจ้งเหตุใหม่ | ตั๋วเพิ่งถูกเปิด ยังไม่ได้กำหนดเส้นทาง | |
| ESCALATED_T2 | ส่งต่อให้ Tier 2 | ส่งต่อให้นักวิเคราะห์ระดับ 2 พิจารณา | |
| T1_REVIEW | รอ Tier 1 ทบทวน | ระดับ 2 ยืนยันว่าเป็นเหตุจริงแล้วส่งกลับให้ระดับ 1 | |
| AWAITING_CONTAINMENT | รอการจัดการจากผู้ดูแลระบบ | มอบหมายให้ผู้ดูแลระบบเข้าควบคุมเหตุการณ์ | |
| CONTAINMENT_REPORTED | รายงานการควบคุมแล้ว | ผู้ดูแลระบบส่งรายงานการควบคุมกลับมาให้ระดับ 1 ตรวจ | |
| PENDING_MANAGER | รอผู้จัดการตรวจสอบ | ส่งให้ผู้จัดการ SOC ตรวจสอบและอนุมัติ | |
| APPROVED | อนุมัติแล้ว | เหตุการณ์ได้รับการจัดการและปิดงานเรียบร้อย | ✓ |
| CLOSED_EVENT | ปิด (Event) | วินิจฉัยว่าไม่ใช่ภัยคุกคามจริง จึงปิดงาน | ✓ |

> สถานะ "อนุมัติแล้ว" และ "ปิด (Event)" คือสถานะสิ้นสุด เมื่อตั๋วงานเข้าสู่สถานะใดสถานะหนึ่งนี้แล้ว จะไม่สามารถดำเนินการต่อได้อีก (ยกเว้นการปรับสถานะฉุกเฉิน ดูหัวข้อแฟล็กฉุกเฉิน)

### ตารางการเปลี่ยนสถานะที่อนุญาต

| จากสถานะ | ไปสถานะ | ผู้ที่ดำเนินการได้ | เงื่อนไข |
|----------|---------|--------------------|----------|
| แจ้งเหตุใหม่ | ปิด (Event) | นักวิเคราะห์ระดับ 1 (ผู้เปิดตั๋ว) | จัดประเภทเป็น Event |

| แจ้งเหตุใหม่ | ส่งต่อให้ Tier 2 | นักวิเคราะห์ระดับ 1 (ผู้เปิดตั๋ว) | จัดประเภทเป็น Incident แล้วขอความเห็นระดับ 2 |
| ส่งต่อให้ Tier 2 | ปิด (Event) | นักวิเคราะห์ระดับ 2 | ระดับ 2 วินิจฉัยใหม่ว่าเป็น Event |
| ส่งต่อให้ Tier 2 | รอ Tier 1 ทบทวน | นักวิเคราะห์ระดับ 2 | ระดับ 2 ยืนยันว่าเป็น Incident ส่งกลับระดับ 1 |
| รอ Tier 1 ทบทวน | รอการจัดการจากผู้ดูแลระบบ | นักวิเคราะห์ระดับ 1 (ผู้เปิดตั๋ว) | มอบหมายผู้ดูแลระบบ |
| รอการจัดการจากผู้ดูแลระบบ | รายงานการควบคุมแล้ว | ผู้ดูแลระบบที่รับผิดชอบ | ส่งรายงานการควบคุม |
| รายงานการควบคุมแล้ว | รอการจัดการจากผู้ดูแลระบบ | นักวิเคราะห์ระดับ 1 (ผู้เปิดตั๋ว) | ยังควบคุมไม่สำเร็จ ส่งกลับให้แก้ไข |
| รายงานการควบคุมแล้ว | รอผู้จัดการตรวจสอบ | นักวิเคราะห์ระดับ 1 (ผู้เปิดตั๋ว) | ควบคุมสำเร็จ และต้องให้ผู้จัดการตรวจสอบ |
| รายงานการควบคุมแล้ว | อนุมัติแล้ว | นักวิเคราะห์ระดับ 1 (ผู้เปิดตั๋ว) | ควบคุมสำเร็จ และไม่ต้องให้ผู้จัดการตรวจสอบ |
| รอผู้จัดการตรวจสอบ | อนุมัติแล้ว | ผู้จัดการ SOC | ผู้จัดการตรวจสอบและอนุมัติปิดงาน |

> เส้นทางจาก "รายงานการควบคุมแล้ว" จะแยกเป็นสองทางโดยอัตโนมัติ ตั๋วที่ต้องผ่านผู้จัดการจะถูกส่งไป "รอผู้จัดการตรวจสอบ" ส่วนตั๋วที่ไม่ต้อง นักวิเคราะห์ระดับ 1 จะปิดได้เลย ระบบเป็นผู้ตัดสินเส้นทางนี้ตามกฎ ไม่สามารถเลือกข้ามได้ (ดูหัวข้อแฟล็กฉุกเฉินและการตรวจสอบโดยผู้จัดการ)

# 2. คู่มือสำหรับนักวิเคราะห์ SOC (Analyst Guide)

ส่วนนี้สำหรับนักวิเคราะห์ SOC ทั้งระดับ 1 (Tier 1) และระดับ 2 (Tier 2) ซึ่งเป็นผู้ปฏิบัติงานหลักในการรับมือเหตุการณ์ประจำวัน

## สรุปบทบาทและความรับผิดชอบ

**นักวิเคราะห์ระดับ 1 (Tier 1)** เป็นด่านแรกและเป็นเจ้าของตั๋วงานตลอดสายงานฝั่งต้นทาง คุณมีหน้าที่:

- รับและคัดแยกการแจ้งเตือน (Triage) ทั้งจากระบบตรวจจับอัตโนมัติและจากการแจ้งด้วยตนเอง
- เปิดตั๋วงานใหม่ และจัดประเภทเหตุการณ์ว่าเป็น Event หรือ Incident
- ตัดสินใจเส้นทางของเหตุการณ์ที่เป็น Incident (มอบหมายผู้ดูแลระบบเอง หรือส่งต่อให้ระดับ 2)
- ตรวจสอบรายงานการควบคุมที่ผู้ดูแลระบบส่งกลับมา และตัดสินว่าควบคุมสำเร็จหรือต้องส่งกลับไปแก้ไข
- ปิดงาน หรือส่งให้ผู้จัดการตรวจสอบ

> ⚠️ ตั๋วงานหนึ่งใบจะผูกกับนักวิเคราะห์ระดับ 1 ที่เป็น **ผู้เปิดตั๋ว** เท่านั้น การดำเนินการฝั่งระดับ 1 ในตั๋วใบนั้น (เช่น การตรวจรายงานการควบคุม การปิดงาน) ทำได้โดยผู้เปิดตั๋วคนเดิมเสมอ เพื่อนร่วมทีมระดับ 1 คนอื่นจะไม่สามารถดำเนินการต่อในตั๋วของคุณได้

**นักวิเคราะห์ระดับ 2 (Tier 2)** เป็นผู้เชี่ยวชาญที่รับเรื่องที่ระดับ 1 ส่งต่อขึ้นมา คุณมีทางเลือกเพียง 2 ทางสำหรับตั๋วที่ถูกส่งต่อ:

- วินิจฉัยว่าเหตุการณ์ไม่ใช่ภัยจริง แล้วปิดงานเป็น Event
- ยืนยันว่าเป็นเหตุการณ์จริง (Incident) แล้วส่งกลับให้ระดับ 1 ดำเนินการต่อ

> ⚠️ ระดับ 2 ไม่สามารถมอบหมายผู้ดูแลระบบเองได้ และไม่สามารถเปิดตั๋วใหม่ได้ การเปิดตั๋วเป็นหน้าที่ของระดับ 1 เท่านั้น

## การเปิดตั๋วงาน (เฉพาะนักวิเคราะห์ระดับ 1)

การเปิดตั๋วงานทำได้เฉพาะนักวิเคราะห์ระดับ 1 มีจุดเริ่มต้นได้ 3 ทาง

### ก. เปิดจากการแจ้งเตือนของระบบตรวจจับ (Wazuh)

การแจ้งเตือนส่วนใหญ่มาจาก **ระบบตรวจจับภัยคุกคาม (Wazuh)** ซึ่งเป็นระบบเฝ้าระวังอัตโนมัติ การแจ้งเตือนเหล่านี้จะปรากฏที่เมนู **Wazuh Triage**

1. เปิดเมนู **Wazuh Triage** คุณจะเห็นรายการแจ้งเตือนที่รอการคัดแยก เรียงตามระดับความรุนแรง
2. กดปุ่ม **รับเรื่อง (Claim)** ที่การแจ้งเตือนที่ต้องการ เพื่อจองว่าคุณเป็นผู้ดูแลรายการนี้ (ป้องกันไม่ให้คนอื่นทำซ้ำ)
3. เมื่อรับเรื่องแล้ว คุณมีทางเลือก 2 ทาง คือ **สร้างตั๋วงาน** หรือ **คืนรายการกลับเข้าคิว** (ต้องระบุเหตุผล)
4. หากเลือกสร้างตั๋วงาน ระบบจะนำคุณไปยังแบบฟอร์มเปิดตั๋ว โดยกรอกข้อมูลบางส่วนให้อัตโนมัติจากการแจ้งเตือน รวมถึงประเมินระดับความรุนแรงเบื้องต้นจากระดับของการแจ้งเตือน

> ระบบจะแปลงระดับการแจ้งเตือนของ Wazuh เป็นระดับความรุนแรงเบื้องต้นให้อัตโนมัติ (ระดับสูงมากเป็น Critical, ระดับสูงเป็น High, ระดับปานกลางเป็น Medium, ที่เหลือเป็น Low) คุณสามารถปรับแก้ได้ตามดุลพินิจก่อนบันทึก

[ภาพหน้าจอ: หน้า Wazuh Triage แสดงรายการแจ้งเตือนพร้อมปุ่มรับเรื่องและระดับความรุนแรง]

### ข. เปิดจากการคัดแยกด้วยตนเอง (Manual Triage)

เหตุการณ์ที่แจ้งเข้ามาทางช่องทางอื่น เช่น อีเมล โทรศัพท์ การแจ้งจากผู้ใช้ภายใน หรือหน่วยงานภายนอก จะบันทึกที่เมนู **Manual Triage** เพื่อเข้าคิวรอการคัดแยก จากนั้นจึงรับรายการและสร้างตั๋วงานในลักษณะเดียวกัน

### ค. เปิดตั๋วใหม่โดยตรง

คุณสามารถเปิดตั๋วใหม่ได้ทันทีจากเมนู **เปิดเคสใหม่** โดยไม่ต้องอ้างอิงการแจ้งเตือนใด

### ขั้นตอนการกรอกแบบฟอร์มเปิดตั๋ว

ในแบบฟอร์มเปิดตั๋ว ให้ดำเนินการดังนี้

1. **จัดประเภทเหตุการณ์ (การจัดประเภท Event/Incident)** — เลือกอย่างใดอย่างหนึ่ง โดยทุกตั๋วต้องระบุค่านี้เสมอ
   - **Event (ไม่เป็นภัย)** — เหตุการณ์ที่ตรวจสอบแล้วว่าไม่ใช่ภัยคุกคามจริง เมื่อบันทึก ตั๋วจะถูกปิดเป็น Event ทันที
   - **Incident (เหตุการณ์จริง)** — เหตุการณ์ที่เป็นภัยจริงและต้องดำเนินการต่อ
2. **เลือกการดำเนินการสำหรับ Incident** — หากเลือกเป็น Incident ต้องเลือกเส้นทางต่อ
   - **มอบหมายให้ผู้ดูแลระบบ (System Admin)** — เมื่อคุณมั่นใจในการวินิจฉัย เลือกผู้ดูแลระบบที่รับผิดชอบ แล้วตั๋วจะถูกส่งไปยังขั้น "รอการจัดการจากผู้ดูแลระบบ"
   - **ส่งต่อให้ Tier 2** — เมื่อยังไม่แน่ใจ ต้องการให้ระดับ 2 ช่วยวินิจฉัย
3. **กรอกรายละเอียดเหตุการณ์** — ระดับความรุนแรง วันและเวลาที่ตรวจพบ ระบบ/บริการที่เกี่ยวข้อง รายละเอียดเหตุการณ์ IP Address ของทรัพย์สิน และข้อมูลอื่น ๆ ตามที่ปรากฏในฟอร์ม
4. **ระบุเจ้าของระบบ (System Owner)** — เลือกหน่วยงาน/เจ้าของระบบที่ได้รับผลกระทบ (หากทราบ) เพื่อให้ระบบแจ้งเตือนเจ้าของระบบโดยอัตโนมัติ
5. **แนบหลักฐาน (ถ้ามี)** — แนบไฟล์ประกอบ เช่น ภาพหน้าจอ บันทึก log โดยไฟล์แต่ละไฟล์มีขนาดไม่เกิน 25 MB
6. กดบันทึกเพื่อเปิดตั๋ว ระบบจะสร้างหมายเลขตั๋วให้อัตโนมัติ

[ภาพหน้าจอ: แบบฟอร์มเปิดตั๋วงาน แสดงตัวเลือกจัดประเภท Event/Incident และตัวเลือกเส้นทาง]

> ⚠️ การจัดประเภทเป็น Event/Incident เป็นการตัดสินใจที่สำคัญ เพราะกำหนดเส้นทางทั้งหมดของตั๋ว หากจัดเป็น Event ตั๋วจะปิดทันที หากไม่แน่ใจ ให้เลือก Incident แล้วส่งต่อให้ระดับ 2 ช่วยวินิจฉัย

## สถานะของตั๋วงานในทางปฏิบัติ

ตารางนี้อธิบายว่าแต่ละสถานะหมายถึงอะไร เกิดจากอะไร และคุณควรทำอะไรต่อ

### แจ้งเหตุใหม่ (NEW)
ตั๋วเพิ่งถูกเปิดและยังไม่ได้กำหนดเส้นทาง โดยปกติเป็นสถานะชั่วคราวที่ระบบจะเดินหน้าต่อทันทีตามการจัดประเภทและเส้นทางที่คุณเลือกในแบบฟอร์มเปิดตั๋ว

### ส่งต่อให้ Tier 2 (ESCALATED_T2)
นักวิเคราะห์ระดับ 1 ส่งเหตุการณ์ที่เป็น Incident ให้ระดับ 2 พิจารณา ตั๋วจะปรากฏในคิวของระดับ 2
- **ระดับ 2 ต้องทำ:** เปิดตั๋ว ตรวจสอบรายละเอียด แล้วเลือกปิดเป็น Event หรือยืนยันเป็น Incident เพื่อส่งกลับระดับ 1

### รอ Tier 1 ทบทวน (T1_REVIEW)
ระดับ 2 ยืนยันแล้วว่าเป็นเหตุการณ์จริง และส่งตั๋วกลับมาให้ระดับ 1 (ผู้เปิดตั๋วเดิม)
- **ระดับ 1 ต้องทำ:** ทบทวนแล้วมอบหมายผู้ดูแลระบบที่รับผิดชอบ พร้อมบันทึกเหตุผล

### รอการจัดการจากผู้ดูแลระบบ (AWAITING_CONTAINMENT)
ตั๋วถูกมอบหมายให้ผู้ดูแลระบบเข้าควบคุมเหตุการณ์แล้ว ระบบส่งอีเมลแจ้งผู้ดูแลระบบโดยอัตโนมัติ
- **ผู้ดูแลระบบต้องทำ:** เข้าควบคุมเหตุการณ์ที่ตัวระบบ แล้วส่งรายงานการควบคุมกลับมา
- **ระดับ 1:** รอรายงานจากผู้ดูแลระบบ

### รายงานการควบคุมแล้ว (CONTAINMENT_REPORTED)
ผู้ดูแลระบบส่งรายงานการควบคุมกลับมาแล้ว ระบบส่งอีเมลแจ้งนักวิเคราะห์ผู้เปิดตั๋ว
- **ระดับ 1 (ผู้เปิดตั๋ว) ต้องทำ:** อ่านรายงาน แล้วตัดสินใจหนึ่งในสามทาง
  1. ยังควบคุมไม่สำเร็จ → ส่งกลับให้ผู้ดูแลระบบแก้ไข พร้อมระบุสิ่งที่ต้องแก้
  2. ควบคุมสำเร็จ และต้องให้ผู้จัดการตรวจสอบ → ส่งให้ผู้จัดการ SOC
  3. ควบคุมสำเร็จ และไม่ต้องให้ผู้จัดการตรวจสอบ → ปิดงานได้เลย

### รอผู้จัดการตรวจสอบ (PENDING_MANAGER)
ตั๋วถูกส่งให้ผู้จัดการ SOC ตรวจสอบขั้นสุดท้าย
- **ผู้จัดการต้องทำ:** ตรวจสอบและอนุมัติปิดงาน

### อนุมัติแล้ว (APPROVED) — สถานะปิดงาน
เหตุการณ์ได้รับการจัดการครบถ้วนและปิดงานเรียบร้อย ระบบส่งอีเมลแจ้งเจ้าของระบบ ไม่มีการดำเนินการต่อ

### ปิด (Event) (CLOSED_EVENT) — สถานะปิดงาน
เหตุการณ์ถูกวินิจฉัยว่าไม่ใช่ภัยคุกคามจริง จึงปิดงาน ระบบส่งอีเมลแจ้งเจ้าของระบบ (หากระบุไว้)

## การเดินหน้าตั๋วงานจากเปิดจนปิด (ตัวอย่างเส้นทาง Incident เต็มรูปแบบ)

สมมติว่ามีการแจ้งเตือนพบมัลแวร์บนเครื่องเซิร์ฟเวอร์ เส้นทางการทำงานอาจเป็นดังนี้

1. **เปิดตั๋ว** — นักวิเคราะห์ระดับ 1 รับการแจ้งเตือนจาก Wazuh Triage สร้างตั๋ว จัดประเภทเป็น **Incident** และเลือก **มอบหมายให้ผู้ดูแลระบบ** พร้อมเลือกผู้ดูแลระบบที่รับผิดชอบ
2. **ผู้ดูแลระบบเข้าควบคุม** — ผู้ดูแลระบบได้รับอีเมลแจ้ง เข้าควบคุมเหตุการณ์ (เช่น แยกเครื่องออกจากเครือข่าย ลบมัลแวร์) แล้วเปิดตั๋วเพื่อ **ส่งรายงานการควบคุม**
3. **ระดับ 1 ตรวจรายงาน** — นักวิเคราะห์ผู้เปิดตั๋วได้รับอีเมลแจ้ง อ่านรายงาน และพบว่าควบคุมเรียบร้อย
4. **ตัดสินเส้นทางปิดงาน**
   - หากเหตุการณ์ระดับ Critical หรือถูกตั้งสถานะฉุกเฉิน → ระบบบังคับให้ **ส่งให้ผู้จัดการตรวจสอบ** แล้วผู้จัดการเป็นผู้อนุมัติปิด
   - หากไม่เข้าเงื่อนไขข้างต้น → นักวิเคราะห์ระดับ 1 **ปิดงาน (อนุมัติ)** ได้เลย
5. **ปิดงาน** — ตั๋วเข้าสู่สถานะ "อนุมัติแล้ว" ระบบแจ้งเจ้าของระบบทางอีเมลโดยอัตโนมัติ

> หากในขั้นตอนที่ 3 พบว่าการควบคุมยังไม่เรียบร้อย ให้ส่งตั๋วกลับไปยังผู้ดูแลระบบพร้อมระบุสิ่งที่ต้องแก้ไข วงจรนี้สามารถวนซ้ำได้จนกว่าจะควบคุมสำเร็จ

[ภาพหน้าจอ: หน้ารายละเอียดตั๋วงาน แสดงปุ่มการดำเนินการตามสถานะปัจจุบันและช่องบันทึกการดำเนินการ]

> ⚠️ ทุกครั้งที่เปลี่ยนสถานะ ระบบจะบังคับให้กรอก **บันทึกการดำเนินการ** เสมอ บันทึกนี้จะถูกเก็บไว้ในประวัติของตั๋วเพื่อการตรวจสอบย้อนหลัง กรุณาเขียนให้ชัดเจนและกระชับ

## แฟล็กฉุกเฉิน (Emergency Flag)

**แฟล็กฉุกเฉิน (Emergency Flag)** คือเครื่องหมายพิเศษที่ติดบนตั๋วงานเพื่อบ่งชี้ว่าเหตุการณ์นี้ต้องการความสนใจเร่งด่วนเป็นกรณีพิเศษ

### ใช้เมื่อใด
ใช้เมื่อเหตุการณ์มีความร้ายแรงหรือเร่งด่วนเกินกว่าระดับความรุนแรงปกติจะสะท้อนได้ เช่น เหตุการณ์ที่อาจส่งผลกระทบเป็นวงกว้างหรือกระทบบริการสำคัญ และคุณต้องการให้ผู้จัดการ SOC เข้ามาตรวจสอบด้วยตนเอง

### ทำให้เกิดอะไร
เมื่อตั้งสถานะฉุกเฉิน ตั๋วงานนั้นจะถูกบังคับให้ต้องผ่านการตรวจสอบจากผู้จัดการ SOC ก่อนปิดเสมอ (แม้ระดับความรุนแรงจะไม่ถึง Critical ก็ตาม) นอกจากนี้ตั๋วยังสามารถกรองและจัดเรียงเพื่อแสดงรายการฉุกเฉินขึ้นก่อนได้

### ใครตั้งได้
- ผู้ดูแลระบบ ผู้จัดการ และระดับ 2 สามารถตั้งหรือยกเลิกได้ทุกเมื่อ
- **นักวิเคราะห์ระดับ 1** ตั้งหรือยกเลิกได้ **เฉพาะตั๋วที่เคยถูกส่งต่อให้ระดับ 2 มาแล้วเท่านั้น** หากตั๋วยังไม่เคยส่งต่อให้ระดับ 2 ระดับ 1 จะไม่สามารถตั้งสถานะฉุกเฉินได้

### ตั้งอย่างไร
1. เปิดหน้ารายละเอียดตั๋วงาน
2. กดปุ่มสลับสถานะฉุกเฉิน (Emergency) พร้อมบันทึกเหตุผลประกอบ (แนะนำให้ระบุเสมอ)
3. ระบบจะบันทึกการเปลี่ยนแปลงลงในประวัติของตั๋วโดยอัตโนมัติ ว่าใครเป็นผู้ตั้ง/ยกเลิก และเมื่อใด

> สถานะฉุกเฉินสามารถปรับเปลี่ยนได้ทุกขั้นตอน รวมถึงตั๋วที่ปิดงานแล้ว

## กระบวนการลงนามรับรอง (Sign-off)

การลงนามรับรองคือการยืนยันความรับผิดชอบในแต่ละขั้นสำคัญ ระบบบันทึกผู้ลงนามและเวลาไว้โดยอัตโนมัติ และ **บันทึกได้ครั้งเดียวเท่านั้น** (เขียนทับไม่ได้)

ระบบมีจุดลงนาม 2 จุด

1. **การรับรองการตรวจสอบ (Verified)** — เกิดขึ้นเมื่อนักวิเคราะห์ระดับ 1 ตัดสินว่าการควบคุมเสร็จสมบูรณ์ (ไม่ว่าจะส่งให้ผู้จัดการต่อ หรือปิดงานเอง) ระบบจะบันทึกชื่อผู้ตรวจสอบและเวลาทันที
2. **การอนุมัติปิดงาน (Approved)** — เกิดขึ้นเมื่อตั๋วเข้าสู่สถานะ "อนุมัติแล้ว" ระบบบันทึกชื่อผู้อนุมัติและเวลา ผู้อนุมัติอาจเป็นนักวิเคราะห์ระดับ 1 (กรณีปิดเองได้) หรือผู้จัดการ SOC (กรณีต้องผ่านการตรวจสอบ)

> การลงนามทั้งสองจุดเกิดขึ้นโดยอัตโนมัติเมื่อคุณดำเนินการเปลี่ยนสถานะตามขั้นตอน คุณไม่ต้องกดปุ่มลงนามแยกต่างหาก เพียงดำเนินการตามขั้นตอน ระบบจะบันทึกชื่อของคุณเป็นผู้รับผิดชอบให้เอง

## การแจ้งเตือนทางอีเมล

ระบบส่งอีเมลแจ้งเตือนอัตโนมัติเมื่อเกิดเหตุการณ์สำคัญ ตารางนี้สรุปว่าเหตุการณ์ใดส่งอีเมลถึงใคร

| เหตุการณ์ | ผู้รับอีเมล | เนื้อหา |
|-----------|------------|---------|
| ตั๋วถูกมอบหมายให้ผู้ดูแลระบบ (เข้าสู่ "รอการจัดการ") | ผู้ดูแลระบบที่รับผิดชอบ | แจ้งว่ามีงานควบคุมที่ต้องดำเนินการ พร้อมลิงก์เข้าตั๋ว |
| ตั๋วถูกส่งกลับให้ผู้ดูแลระบบแก้ไข | ผู้ดูแลระบบที่รับผิดชอบ | แจ้งว่าต้องแก้ไขการควบคุม พร้อมเหตุผลจากนักวิเคราะห์ |
| ผู้ดูแลระบบส่งรายงานการควบคุม | นักวิเคราะห์ผู้เปิดตั๋ว | แจ้งว่ามีรายงานรอตรวจสอบ |
| เปิดตั๋วงานใหม่ | เจ้าของระบบ | แจ้งว่าพบเหตุการณ์ความปลอดภัยบนระบบ |
| ปิดตั๋วงาน (อนุมัติ หรือปิด Event) | เจ้าของระบบ | แจ้งผลการดำเนินการ |

**สิ่งที่นักวิเคราะห์ควรคาดหวัง:** ในฐานะนักวิเคราะห์ระดับ 1 คุณจะได้รับอีเมลเมื่อผู้ดูแลระบบส่งรายงานการควบคุมกลับมา ซึ่งเป็นสัญญาณว่าถึงคิวคุณต้องเข้าตรวจสอบ

> ⚠️ อีเมลแจ้งเตือนทุกฉบับเป็นการแจ้งทางเดียว ห้ามตอบกลับ และให้เข้าดำเนินการในระบบโดยตรงผ่านลิงก์ในอีเมลเสมอ หากระบบส่งอีเมลไม่สำเร็จ จะมีข้อความเตือนปรากฏบนหน้าจอ ให้ติดต่อผู้เกี่ยวข้องด้วยตนเอง

## ความเข้าใจเรื่อง SLA

**SLA คืออะไร:** ระบบกำหนดกรอบเวลามาตรฐานไว้ที่ **4 ชั่วโมง** นับจากเวลาที่ตรวจพบเหตุการณ์ ระบบใช้เวลานี้คำนวณ **เส้นตาย (SLA Deadline)** ของแต่ละตั๋ว

**ระบบติดตามอะไร:** นาฬิกา SLA วัดว่าทีมตอบสนองและเปิดตั๋วได้ทันภายในกรอบเวลาหรือไม่ ระบบจะแสดงเครื่องหมายเตือนเมื่อตั๋วใกล้ครบกำหนด (เหลือเวลาน้อยกว่า 1 ชั่วโมง) และเมื่อเกินกำหนด

**การเกินกำหนด (Breach) หมายถึงอะไร:** หมายถึงเหตุการณ์ไม่ได้รับการตอบสนองภายในกรอบเวลาที่กำหนด ซึ่งเป็นตัวชี้วัดที่ผู้บริหารติดตาม

**ควรทำอย่างไร:**
- ในหน้ารายการตั๋ว (Active Incidents) ให้จัดเรียงตาม SLA เพื่อให้ตั๋วที่ใกล้ครบกำหนดที่สุดขึ้นก่อน
- ให้ความสำคัญกับตั๋วที่มีเครื่องหมายเตือน SLA ก่อนเสมอ
- หากเหตุการณ์มีความเร่งด่วนเป็นพิเศษ พิจารณาใช้แฟล็กฉุกเฉินและประสานงานกับผู้เกี่ยวข้องโดยตรง

[ภาพหน้าจอ: หน้ารายการตั๋วงาน แสดงเครื่องหมายเตือน SLA และตัวกรอง/จัดเรียง]

# 3. คู่มือสำหรับผู้บริหาร (Management Guide)

ส่วนนี้สำหรับผู้จัดการ SOC (SOC Manager) ซึ่งมีหน้าที่กำกับดูแล อนุมัติ และติดตามภาพรวมการทำงานของทีม

## สรุปบทบาทและความรับผิดชอบ

ในฐานะผู้จัดการ SOC คุณมีหน้าที่หลักดังนี้

- **ตรวจสอบและอนุมัติปิดงาน** เหตุการณ์สำคัญที่ถูกส่งมาให้คุณ (สถานะ "รอผู้จัดการตรวจสอบ")
- **ติดตามภาพรวม** ประสิทธิภาพและปริมาณงานของทีมผ่านแดชบอร์ด (Dashboard)
- **กำกับดูแล** เหตุการณ์เร่งด่วนและการเกินกำหนด SLA

> ระบบจะแสดงรายการตั๋วในเมนู Active Incidents ให้ผู้จัดการเห็น **เฉพาะตั๋วที่อยู่ในสถานะ "รอผู้จัดการตรวจสอบ"** เท่านั้น เพื่อให้คุณโฟกัสที่งานที่ต้องตัดสินใจ ส่วนภาพรวมทั้งหมดดูได้ที่แดชบอร์ด

## การดูและกรองตั๋วงาน

1. เปิดเมนู **Active Incidents** เพื่อดูตั๋วที่รอการอนุมัติจากคุณ
2. ใช้เครื่องมือค้นหาและตัวกรองเพื่อจำกัดผลลัพธ์ เช่น
   - **ค้นหา** ด้วยหมายเลขตั๋ว ชื่อระบบ หรือ IP
   - **กรองตามระดับความรุนแรง** (Critical / High / Medium / Low)
   - **กรองตามสถานะฉุกเฉิน**
   - **จัดเรียง** ตาม SLA ความเร่งด่วน หรือวันที่
3. เปิดเมนู **ประวัติ Ticket** เพื่อดูตั๋วที่ปิดงานแล้ว พร้อมตัวกรองตามช่วงวันที่ ผู้อนุมัติ สถานะการปิด และระดับความรุนแรง

[ภาพหน้าจอ: หน้ารายการตั๋วงานของผู้จัดการ แสดงตัวกรองและตั๋วที่รออนุมัติ]

## ตัวชี้วัดบนแดชบอร์ด

เมนู **Dashboard** สรุปภาพรวมการทำงานของทีมในรูปแบบที่อ่านและนำไปใช้ตัดสินใจได้ทันที คุณสามารถปรับช่วงเวลา (วันนี้ / สัปดาห์นี้ / เดือนนี้ / ทั้งหมด) ระดับความรุนแรง และสถานะ เพื่อเจาะดูข้อมูลได้

[ภาพหน้าจอ: หน้าแดชบอร์ดภาพรวม แสดงการ์ดตัวชี้วัดและกราฟต่าง ๆ]

### เวลาเฉลี่ยในการแก้ไข (MTTR)
**เวลาเฉลี่ยในการแก้ไข (Mean Time To Resolve — MTTR)** คือระยะเวลาเฉลี่ยตั้งแต่เปิดตั๋วจนปิดงาน คำนวณจากตั๋วที่ปิดในรอบ 30 วันที่ผ่านมา แสดงทั้งค่ามัธยฐาน (median) และค่าเฉลี่ย (mean) เป็นหน่วยชั่วโมง
- **อ่านอย่างไร:** ค่ายิ่งต่ำ แสดงว่าทีมจัดการเหตุการณ์ได้เร็ว หากค่าสูงขึ้นผิดปกติ อาจบ่งชี้ถึงคอขวดในขั้นตอนใดขั้นตอนหนึ่ง
- นอกจากนี้ยังมี **MTTR แยกตามประเภทเหตุการณ์** เพื่อชี้ว่าเหตุการณ์ประเภทใดใช้เวลาจัดการนานที่สุด

### อัตราการปฏิบัติตาม SLA (SLA Compliance)
แสดงเป็นเปอร์เซ็นต์ของตั๋วที่ปิดได้ภายในกรอบเวลา SLA นอกจากนี้ยังมี
- **เกินกำหนดแล้ว (SLA Breach)** — จำนวนตั๋วที่ยังเปิดอยู่และเลยเส้นตายไปแล้ว
- **ใกล้ครบกำหนด (At Risk)** — จำนวนตั๋วที่กำลังจะถึงเส้นตายในอีกไม่กี่ชั่วโมง
- **อ่านอย่างไร:** อัตราการปฏิบัติตาม SLA ที่สูงเป็นเป้าหมาย หากจำนวนตั๋วเกินกำหนดหรือใกล้ครบกำหนดสูง ควรพิจารณาเพิ่มกำลังคนหรือจัดลำดับความสำคัญใหม่

### ปริมาณงาน (Ticket Volume)
- **แนวโน้มรายเดือน/รายวัน** — จำนวนตั๋วที่เปิดในแต่ละช่วงเวลา ช่วยให้เห็นภาระงานและแนวโน้ม
- **ปิดงานเดือนนี้เทียบเดือนก่อน** — เปรียบเทียบจำนวนตั๋วที่ปิดได้
- **การกระจายตามความรุนแรง** และ **ตามประเภท/หมวดหมู่** — ช่วยให้เห็นว่าภัยประเภทใดพบมากที่สุด
- **คงค้างแยกตามอายุงาน (Backlog Aging)** — แบ่งตั๋วที่ยังเปิดอยู่เป็นกลุ่มใหม่ (0–24 ชม.) กำลังค้าง (1–3 วัน) และค้างนาน (เกิน 3 วัน) เพื่อเฝ้าระวังงานที่ค้างนานผิดปกติ

### ตัวชี้วัดเสริมอื่น ๆ
- **เวลาตอบสนอง (Analyst Response Time)** — ระยะเวลาตั้งแต่การแจ้งเตือนพร้อมรับจนเปิดตั๋ว (เฉพาะตั๋วที่มาจากการแจ้งเตือนอัตโนมัติ)
- **ภาระงานรายบุคคล (Assignee Workload / Heatmap)** — จำนวนตั๋วที่เปิดอยู่ในความรับผิดชอบของแต่ละคน พร้อมจำนวนที่เกินกำหนด
- **สัดส่วน Event/Incident** — สัดส่วนเหตุการณ์ที่เป็นภัยจริงเทียบกับที่ไม่เป็นภัย
- **มุมมองสายงาน (Pipeline)** — จำนวนตั๋วในแต่ละสถานะ ช่วยให้เห็นว่างานไปกระจุกที่ขั้นตอนใด

## ความรับผิดชอบด้านการอนุมัติและการลงนาม

ตั๋วงานจะถูกส่งมาให้คุณตรวจสอบ (สถานะ "รอผู้จัดการตรวจสอบ") เมื่อเข้าเงื่อนไขอย่างใดอย่างหนึ่ง

- เหตุการณ์มีระดับความรุนแรงถึงเกณฑ์ที่กำหนด (โดยมาตรฐานคือระดับ **Critical**) หรือ
- ตั๋วถูกตั้ง **สถานะฉุกเฉิน (Emergency)**

ขั้นตอนการอนุมัติ:

1. เปิดตั๋วที่อยู่ในสถานะ "รอผู้จัดการตรวจสอบ"
2. ตรวจสอบรายละเอียดเหตุการณ์ รายงานการควบคุม และประวัติการดำเนินการ
3. หากเห็นชอบ ให้กด **ตรวจสอบและปิดงาน (อนุมัติ)** พร้อมบันทึกการดำเนินการ
4. ตั๋วจะเข้าสู่สถานะ "อนุมัติแล้ว" ระบบบันทึกชื่อคุณเป็นผู้อนุมัติและเวลาโดยอัตโนมัติ และแจ้งเจ้าของระบบทางอีเมล

> การอนุมัติเป็นการลงนามรับรองที่บันทึกครั้งเดียวและเขียนทับไม่ได้ ชื่อของคุณจะปรากฏเป็นผู้อนุมัติในประวัติของตั๋วอย่างถาวร

> นอกจากการอนุมัติแล้ว ผู้จัดการยังสามารถตั้ง/ยกเลิกสถานะฉุกเฉินของตั๋วใดก็ได้ และแก้ไขบันทึกการดำเนินการในประวัติของตั๋วได้ (เนื่องจากเป็นส่วนหนึ่งของการกำกับดูแลร่องรอยการตรวจสอบ)

## การเฝ้าระวังเหตุการณ์เร่งด่วน

เหตุการณ์ฉุกเฉินจะปรากฏให้คุณเห็นได้หลายทาง

- ตั๋วที่ตั้งสถานะฉุกเฉินจะถูกบังคับให้ส่งมาให้คุณตรวจสอบก่อนปิดงานเสมอ จึงปรากฏในรายการ "รอผู้จัดการตรวจสอบ"
- คุณสามารถกรองและจัดเรียงรายการตั๋วเพื่อแสดงเหตุการณ์ฉุกเฉินขึ้นก่อนได้
- แดชบอร์ดแสดงตั๋วระดับ Critical ที่ยังเปิดอยู่ และตั๋ว Critical ที่ใกล้ครบกำหนด SLA มากที่สุด

**สิ่งที่ผู้จัดการควรทำ:** เมื่อพบเหตุการณ์เร่งด่วนหรือใกล้เกินกำหนด ควรเข้าตรวจสอบโดยเร็ว ติดตามกับนักวิเคราะห์ผู้รับผิดชอบ และตัดสินใจอนุมัติหรือให้คำแนะนำเพื่อเร่งการดำเนินการ

## สิ่งที่ผู้จัดการไม่สามารถแก้ไขได้

เพื่อป้องกันความสับสน โปรดทราบว่าระบบออกแบบให้แต่ละบทบาทรับผิดชอบเฉพาะส่วนของตน ผู้จัดการ SOC **ไม่สามารถ** ดำเนินการต่อไปนี้

- **เปิดตั๋วงานใหม่** — เป็นหน้าที่ของนักวิเคราะห์ระดับ 1 เท่านั้น
- **คัดแยกการแจ้งเตือน (Triage)** จากระบบตรวจจับ — เป็นหน้าที่ของนักวิเคราะห์ระดับ 1
- **จัดประเภทเหตุการณ์เป็น Event/Incident** — เป็นการตัดสินใจของระดับ 1 (หรือระดับ 2 กรณีถูกส่งต่อ)
- **มอบหมายผู้ดูแลระบบ** — เป็นหน้าที่ของระดับ 1
- **ส่งรายงานการควบคุม** — เป็นหน้าที่ของผู้ดูแลระบบที่รับผิดชอบ

หน้าที่ในสายงานหลักของผู้จัดการ คือการ **ตรวจสอบและอนุมัติปิดงาน** ตั๋วที่ถูกส่งมาให้เท่านั้น ส่วนการกำกับดูแลทำผ่านแดชบอร์ดและการตั้งสถานะฉุกเฉิน

> หากคุณต้องการให้มีการดำเนินการในขั้นตอนที่ไม่ใช่หน้าที่ของคุณ กรุณาประสานงานกับนักวิเคราะห์หรือผู้ดูแลระบบผู้รับผิดชอบโดยตรง แทนการพยายามแก้ไขเอง
